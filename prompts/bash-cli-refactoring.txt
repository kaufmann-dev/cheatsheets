Refactor the existing Git Wrangler Bash scripts into a cohesive CLI tool with a dispatcher and help system.

Goal:

- do NOT change any functional code or logic inside the existing scripts
- use a `libexec/` directory to store all subcommand scripts
- rename all `.sh` scripts to the `wrangler-<subcommand>` naming convention (no extension)
- each subcommand script must have a standardized header block delimited by `# ====` containing `Usage`, `Description`, and `Category`
- category must be a logical grouping label like "Remote Operations", "Local Operations", or "Utility"
- the top-level `wrangler` dispatcher must accept no flags of its own
- help must be handled exclusively through `wrangler help`

Process:
1. Create a `libexec/` directory and move/rename all existing `.sh` scripts into it (e.g., `wrangler-pull`).
2. Update the header blocks inside each renamed script to include the `Usage`, `Description`, and `Category` fields.
3. Create the top-level `wrangler` dispatcher script that dynamically resolves its installation directory.
4. Intercept the first argument as the subcommand in the dispatcher, defaulting to "help" if none is provided.
5. Validate the subcommand to prevent directory traversal (`/` or `\`) and execute it via `exec bash`.
6. Create `libexec/wrangler-help` to dynamically discover subcommands and parse their headers for the top-level help menu.
7. Implement per-subcommand documentation extraction in `wrangler-help` using `awk`.
8. Rewrite `README.md` and `AGENTS.md` to document the new `wrangler <subcommand>` architecture.

Output:
Provide the shell commands and file contents to execute this refactoring.

Do not use `.sh` extensions for scripts in `libexec/`.
Do not implement `--help` or `-h` flags inside individual subcommands.
